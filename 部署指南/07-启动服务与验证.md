# 第七册：启动服务与验证

> **本册在哪操作**：**云服务器** SSH 命令行。
>
> **目标**：把 API 服务和爬虫服务注册到 systemd（开机自启、自动拉起），启动两个服务，然后做全链路验证。

---

## 1. 先理解项目的两个后端服务

项目后端拆成两个独立的系统服务，职责分开：

| systemd 服务名 | 脚本 | 职责 | 监听端口 |
|---|---|---|---|
| `rejubang-api` | `gunicorn ... run:app` | Flask API 服务（处理小程序请求） | 127.0.0.1:5000（只内网） |
| `rejubang-crawler` | `python scheduler/task_scheduler.py` | 定时爬虫 + 数据处理 | 不监听端口 |

两个服务都在 `/opt/rejubang/backend/` 目录下运行，都用同一个 `venv/` 虚拟环境，都从同一个 `.env` 读取配置。

---

## 2. 创建日志目录

```
root@server:~# mkdir -p /opt/rejubang/logs
root@server:~# ls -ld /opt/rejubang/logs
drwxr-xr-x 2 root root ... /opt/rejubang/logs
```

爬虫服务启动后，日志会写到 `/opt/rejubang/logs/scheduler_YYYY-MM-DD.log`（按天分文件）。

---

## 3. 安装 systemd 服务文件

项目自带了两个 `.service` 文件在 `/opt/rejubang/deploy/` 下。

### 3.1 切到 deploy 目录

```
root@server:~# cd /opt/rejubang/deploy
root@server:/opt/rejubang/deploy# ls *.service
rejubang-api.service  rejubang-crawler.service
```

### 3.2 复制到 systemd 系统目录

```
root@server:/opt/rejubang/deploy# cp rejubang-api.service /etc/systemd/system/
root@server:/opt/rejubang/deploy# cp rejubang-crawler.service /etc/systemd/system/
```

### 3.3 让 systemd 重新加载配置

```
root@server:/opt/rejubang/deploy# systemctl daemon-reload
```

---

## 4. 设置开机自启并启动

### 4.1 API 服务

```
root@server:~# systemctl enable --now rejubang-api
```

`enable --now` = `enable`（开机自启）+ `start`（立即启动）。

### 4.2 爬虫服务

```
root@server:~# systemctl enable --now rejubang-crawler
```

---

## 5. 检查两个服务都在运行

### 5.1 查看 API 服务状态

```
root@server:~# systemctl status rejubang-api --no-pager
● rejubang-api.service - RejuBang API Service
     Loaded: loaded (/etc/systemd/system/rejubang-api.service; enabled; ...)
     Active: active (running) since ...
   Main PID: 12345 (gunicorn)
      Tasks: 3 ...
     Memory: ...
        CPU: ...
     CGroup: /system.slice/rejubang-api.service
             ├─ 12345 /opt/rejubang/backend/venv/bin/python .../gunicorn ...
             ├─ 12346 /opt/rejubang/backend/venv/bin/python .../gunicorn ...
             └─ 12347 /opt/rejubang/backend/venv/bin/python .../gunicorn ...
```

关键看 **`Active: active (running)`**。

### 5.2 查看爬虫服务状态

```
root@server:~# systemctl status rejubang-crawler --no-pager
● rejubang-crawler.service - RejuBang Crawler Service (Playwright)
     ...
     Active: active (running) since ...
```

同样看到 `Active: active (running)` 即可。

### 5.3 如果任一个显示 `failed` 或 `inactive`

先看日志找原因：

```
# API 服务的最近 50 行日志
root@server:~# journalctl -u rejubang-api -n 50 --no-pager

# 爬虫服务的最近 50 行日志
root@server:~# journalctl -u rejubang-crawler -n 50 --no-pager
```

最常见的原因：
- `.env` 里密码错误 → 回 [05-环境变量配置.md](./05-环境变量配置.md) 检查
- 端口 5000 已被占用 → `ss -ltn | grep 5000` 查进程，`kill -9 <PID>` 或改 `gunicorn_config.py`

## 6. 本地验证 API（不走 HTTPS）

API 服务直接在 `127.0.0.1:5000` 监听，可以绕过 Nginx 直接测：

```
root@server:~# curl http://127.0.0.1:5000/health
{"status":"ok"}
```

看到 `{"status":"ok"}` 就说明 Flask 跑起来了。

---

## 7. 完整的外部 HTTPS 验证

在 **服务器** 上 curl 外部域名，测试整个链路（小程序→Nginx→Flask）：

```
root@server:~# curl https://$API_DOMAIN/health
{"status":"ok"}
```

### 7.1 测一个真正的 API

```
root@server:~# curl "https://$API_DOMAIN/api/v1/heat/realtime/rank?limit=3"
```

应该返回 JSON（初始数据库有 30 部种子剧，可能暂时没有热度值，`data` 字段可能是空数组 `[]`，这是正常的——还没跑过第一次采集）：

```json
{"code":0,"msg":"success","data":[]}
```

如果返回 `{"code":500,...}`，看日志：

```
root@server:~# journalctl -u rejubang-api -n 100 --no-pager | tail -50
```

---

## 8. 查看爬虫首次执行情况

爬虫服务启动后会**立刻执行一轮热度采集**，可以实时看日志：

```
root@server:~# tail -f /opt/rejubang/logs/scheduler_$(date +%Y-%m-%d).log
```

（按 **Ctrl + C** 退出 tail）

应该看到：
- `调度器启动` / `scheduler started`
- `开始采集热度` / 各平台的采集日志
- 可能有 `未发现任何剧` 或 `热度值: 0`（平台页面结构变化需要维护爬虫，**不影响 API 正常运行**）

---

## 9. 手动触发一次采集看看数据

### 9.1 切到后端目录

```
root@server:~# cd /opt/rejubang/backend
```

### 9.2 单独测试某一个平台（不保存到数据库）

```
root@server:/opt/rejubang/backend# ./venv/bin/python test_crawl.py --platform tencent
```

可选平台：`tencent` / `iqiyi` / `youku` / `mgtv`。

### 9.3 全平台采集并保存数据库

```
root@server:/opt/rejubang/backend# ./venv/bin/python test_crawl.py --save
```

跑完看数据库里的结果：

```
root@server:/opt/rejubang/backend# mysql -u rejubang -p'i7qxW0ZUGwWBMSUqm2TdJCB3DefTwosA' rejubang -e "
SELECT d.title, p.name AS platform, h.heat_value, h.record_time
FROM heat_realtime h
JOIN dramas d ON h.drama_id = d.id
JOIN platforms p ON h.platform_id = p.id
ORDER BY h.record_time DESC LIMIT 20;
"
```

如果爬虫工作正常，会看到 20 行剧名+热度值+时间的记录。

## 10. 定时任务说明

爬虫服务启动后会自动运行以下 8 个定时任务（**无需人工干预**）：

| 时间（Asia/Shanghai） | 任务 | 说明 |
|---|---|---|
| 每 15 分钟 | 实时热度采集 | Playwright 打开四平台页面提取热度值 |
| 每日 00:00 | 数据清洗 | 去除异常值和重复记录 |
| 每日 00:30 | 日度统计 | 计算每日热度均值/最高/最低 |
| 每日 01:00 | 剧力指数 | 综合热度 35% + 社交 25% + 播放 25% + 口碑 15% |
| 每日 01:30 | 异动检测 | 检测热度暴涨的剧（>200%） |
| 每日 03:00 | 豆瓣评分 | 更新在播剧的豆瓣评分 |
| 每日 04:00 | 旧数据清理 | 删除 30 天前的实时热度和播放量快照 |
| 每日 15:00 | 数据发布 | 把前一日计算完成的日榜数据标记为 published |
